# 🏗️ 软件架构与调试工程化

> [!abstract] 本篇速览
> 分层架构、时间片调度器、FSM、VOFA+/JustFloat 调参

## 总体架构：分层 + 调度 + 状态机

电赛"小车/循迹"类项目代码量虽不大（通常 3k~8k 行），但参与人多、迭代快、临场改参数，**工程化的核心目标是"可联调、可调参、可移植、可甩锅定位"**。推荐统一三件套：**分层目录 + 时间片调度器 + 任务级有限状态机（FSM）**，配合 VOFA+ 在线波形整定。

### 分层架构（BSP/HAL → Driver → Middleware → App）

| 层 | 职责 | MSPM0 对应 | 是否随芯片改动 |
|---|---|---|---|
| **HAL/BSP** | 寄存器/外设抽象 | TI DriverLib（`DL_GPIO_*`/`DL_TimerG_*`/`DL_UART_*`）+ SysConfig 生成的 `ti_msp_dl_config.c` | 换芯片全改 |
| **Driver** | 设备驱动，不含业务 | `motor.c`(PWM+方向)、`encoder.c`(QEI)、`oled_ssd1306.c`、`k230_uart.c`、`imu.c` | 改 HAL 调用 |
| **Middleware** | 通用算法/组件 | `pid.c`、`filter.c`(滑动/一阶低通)、`scheduler.c`、`fsm.c`、`kv_flash.c`(参数存储)、`vofa.c` | 基本不动 |
| **App** | 业务逻辑/比赛流程 | `app_track.c`(循迹)、`app_mission.c`(任务FSM)、`app_comm.c`(K230协议)、`main.c` | 与芯片无关 |

目录建议：`/Bsp /Drivers /Middlewares /App /Components(第三方) /Doc`。**关键纪律**：App 层禁止直接调 `DL_*` 寄存器函数，必须经 Driver；Driver 不允许出现"比赛逻辑"。这样 STM32 练手代码（HAL 库）和 MSPM0 主力代码可以**共用 Middleware/App，只换 BSP/Driver**，移植成本最低。

## 任务调度：裸机时间片轮询 vs RTOS

### 选型结论（电赛场景）

**绝大多数小车/循迹赛题用"前后台 + 时间片轮询"即可，不上 FreeRTOS。** 理由：控制环周期固定（1~5ms）、任务数少（<15）、无复杂阻塞/优先级反转；裸机更易调试（无栈溢出、无任务挂死黑盒），临场加任务只需在调度表加一行。**只有当出现多个独立长耗时阻塞任务（如同时跑文件系统、TCP/IP、复杂图像处理）才考虑 FreeRTOS。** 嵌入式架构三态：轮询系统 → 前后台系统 → 多任务系统（RTOS），电赛停在前两个。

### 时间片轮询调度器（SysTick 心跳，可直接用）

用 SysTick 或一路定时器产生 1ms tick，调度表用结构体数组 + 函数指针，按各自周期触发标志：

```c
typedef struct {
    void (*task)(void);     // 任务函数
    uint16_t period_ms;     // 运行周期
    uint16_t timer;         // 倒计时
    uint8_t  run;           // 就绪标志
} task_t;

static task_t s_tasks[] = {
    {task_pid_ctrl,   2,   0, 0},   // 控制环 500Hz（也可放中断里）
    {task_track,      5,   0, 0},   // 循迹决策 200Hz
    {task_k230_parse, 5,   0, 0},   // 解析 K230 数据
    {task_oled,      50,   0, 0},   // OLED 刷新 20Hz
    {task_vofa_send, 20,   0, 0},   // 上位机回传 50Hz
    {task_key,       20,   0, 0},   // 按键扫描+消抖
    {task_wdg_feed, 100,   0, 0},   // 喂狗
};
#define TASK_NUM (sizeof(s_tasks)/sizeof(task_t))

// SysTick / 1ms 定时器中断里：
void scheduler_tick_1ms(void) {
    for (uint8_t i = 0; i < TASK_NUM; i++)
        if (s_tasks[i].timer == 0) { s_tasks[i].run = 1; s_tasks[i].timer = s_tasks[i].period_ms; }
        else s_tasks[i].timer--;
}
// main while(1) 里：
void scheduler_run(void) {
    for (uint8_t i = 0; i < TASK_NUM; i++)
        if (s_tasks[i].run) { s_tasks[i].run = 0; s_tasks[i].task(); }
}
```

注意：**中断里只置位、不干活**（标志置位 ~µs 级），耗时任务全在主循环跑，避免 tick 中断被拖长导致调度抖动。

## 比赛流程用有限状态机（FSM）

把"出发→循迹→识别→执行→返回→停车"建模为状态机，杜绝 `if-else` 嵌套地狱，**便于现场只改一个状态而不动全局**。表驱动写法最清晰：

```c
typedef enum { ST_IDLE, ST_START, ST_TRACK, ST_DETECT,
               ST_ACTION, ST_RETURN, ST_STOP, ST_ERR } state_e;

static state_e g_state = ST_IDLE;

void mission_fsm(void) {           // 放在 task_mission，5~10ms 调一次
    switch (g_state) {
    case ST_IDLE:   if (key_start())        g_state = ST_START;  break;
    case ST_START:  motor_go(BASE_SPEED);   g_state = ST_TRACK;  break;
    case ST_TRACK:  track_follow();
                    if (k230_target_found()) g_state = ST_DETECT; // K230 给的事件
                    if (crossing_count()>=N)  g_state = ST_RETURN; break;
    case ST_DETECT: if (k230_result_stable(3)) g_state = ST_ACTION; break; // 连续3帧一致
    case ST_ACTION: do_action(); g_state = ST_TRACK; break;
    case ST_RETURN: track_follow(); if (at_home()) g_state = ST_STOP; break;
    case ST_STOP:   motor_stop(); buzzer_beep(); break;
    case ST_ERR:    motor_stop(); g_state = ST_STOP; break;
    }
}
```

工程要点：**状态切换与状态内动作分离**；**进入/退出动作显式写**（如进 ST_ACTION 先停车）；FSM 与 K230 视觉解耦——视觉只产生"事件/结果"，由 FSM 决定何时采信（上面用"连续 N 帧一致"防误识别）。

## 中断与优先级、控制环周期

### Cortex-M0+（MSPM0）优先级特性

MSPM0G3507 是 **Cortex-M0+，NVIC 仅 2 个优先级位 → 只有 4 级（0 最高 ~ 3 最低）**，且**无子优先级、不支持任意嵌套配置**（M0+ 只有抢占级）。用 DriverLib：`NVIC_SetPriority(TIMx_INT_IRQn, 0)` 或 `DL_Interrupt`/CMSIS 接口设置。建议分配：

| 中断源 | 优先级 | 说明 |
|---|---|---|
| 控制环定时器（1~2ms） | 0（最高） | 编码器采样+PID，不可被拖 |
| 编码器/捕获 | 0~1 | 计数即时 |
| K230/串口 RX | 1 | DMA 或 IDLE 中断收帧 |
| SysTick / 调度 tick | 2 | 心跳 |
| VOFA 发送/低速外设 | 3（最低） | |

### 控制环周期与采样

主流取 **1ms（1kHz）或 2ms（500Hz）**：MSPM0L 主频 32MHz、分频 32000 即 1kHz；G3507 可至 80MHz。编码器测速用 **M 法（固定时间窗计数）**：
$$v(\text{rpm}) = \frac{\Delta\text{count}}{PPR \times 4 \times \Delta t} \times 60$$
其中 4 倍频来自正交解码（QEI），$\Delta t$ 即控制环周期。**控制环建议直接放定时器中断里执行**（计算量小、确定性强），调度器只管慢任务。PID 微分项务必对"测量值"求导（防 setpoint 突变踢腿），积分加限幅抗饱和。

## 参数掉电保存、看门狗、上电自检

### MSPM0 Flash 模拟 EEPROM

MSPM0G3507 有 128KB Flash，**低 32KB 区支持 10 万次擦写、其余 1 万次**——参数区放低地址区。方法：链接脚本开一段 `.EEPROM` 扇区，用 DriverLib 的 `DL_FlashCTL_unprotectSector` → `DL_FlashCTL_eraseMemory`(扇区擦除) → `DL_FlashCTL_programMemory*`(编程，建议 `programMemoryFromRAM` / 带 ECC 的接口) 写入。**Flash 写前必须按扇区擦除（最小 1KB/扇区）、地址按 8 字节对齐**。

```c
const uint32_t EEPROM_BUF[256] __attribute__((used,section(".EEPROM"))) = {0};
```

**严禁误写 NONMAIN 区**（存 BSL/BCR 启动配置），写坏会导致芯片无法烧录（E2E 上大量"non-main contents error"翻车案例）。实战封装一个 KV：`param_save()`/`param_load()`，存 PID 三参、基准速度、循迹偏置；加 **magic + CRC 校验**，校验失败回退默认值。

### 看门狗 + 上电自检

MSPM0G 有 **2 个窗口看门狗（WWDT）**；电赛建议普通模式喂狗即可，喂狗放最低优先级慢任务（如 100ms），**绝不能在中断里盲喂**（会掩盖死循环）。上电自检（POST）顺序：时钟/复位源判断 → Flash 参数 CRC → IMU/编码器自检（读 WHO_AM_I、空转测脉冲）→ K230 握手（发 ping 等 ack）→ OLED 显示"SELF-TEST OK"，任一失败进 ST_ERR 并蜂鸣报警。

## 调试与上位机：VOFA+ / 匿名 / OLED 菜单

### VOFA+ JustFloat 协议（强烈推荐，PID 整定神器）

JustFloat 是**小端 float 数组 + 4 字节固定帧尾 `0x00,0x00,0x80,0x7F`** 的字节流协议，适合多通道高频回传。每通道 4 字节 float，连续排列后追加帧尾即可被波形识别：

```c
void vofa_justfloat_send(float *ch, uint8_t n) {
    uint8_t tail[4] = {0x00, 0x00, 0x80, 0x7F};
    uart_send_bytes((uint8_t*)ch, n * 4);   // n 个小端 float
    uart_send_bytes(tail, 4);               // 帧尾
}
// 用法：float d[3]={target,actual,pwm}; vofa_justfloat_send(d,3);
```

另一协议 **FireWater** 是 ASCII（`通道,通道\n`），可读但占带宽；调试用 FireWater，整定 PID 用 JustFloat。VOFA+ 还支持回写指令面板，实现**串口在线整定 PID**（下位机解析 `p=1.2\n` 之类命令改参，不必重新烧录）。**匿名上位机 V7** 是另一选择：帧头 `0xAA`、功能码 `0xF1~0xFA`、和校验(SC)+附加校验(AC) 各取低 8 位，从 `0xAA` 累加到 DATA 末；功能更杂但波形不如 VOFA+ 直观。

### 本地调参：按键 + OLED 菜单

无电脑时靠 **OLED 菜单 + 编码器旋钮/三键（上/下/确认）** 改参：用一个小 FSM 管菜单层级（主菜单→PID 子菜单→数值编辑），短按切项、长按进入/保存，改完写 Flash。这是电赛现场标配，务必提前做好。日志层建议分级（`LOG_E/W/I/D`）经 UART 输出，发布版本用宏关掉 `LOG_D` 省带宽。

## 团队协作与 Git

- **`.gitignore` 屏蔽编译产物与 IDE 缓存**：Keil 的 `*.o *.crf *.d *.lst *.map *.axf *.htm Objects/ Listings/ *.uvguix.*`；CCS/MSPM0 的 `Debug/ *.obj .launches/`。只提交源码 + `.uvprojx`/工程文件 + SysConfig 的 `.syscfg`。
- **分支模型（轻量）**：`main`(可演示稳定版) + 个人/功能分支 `feat-track`、`feat-vision`、`fix-pid`；每天先 `pull`/rebase 主干再干活，减少冲突。**约定一人主要负责一组文件**（电控/视觉/上位机分模块），从根上避免改同一文件。
- **比赛节点打 tag**：`v1.0-track-ok`、`v2.0-final`，临场炸了能秒回退。提交信息写清"改了什么参数/为什么"，比赛后期参数变更频繁，commit 历史就是调参日志。

> 交叉验证来源：TI E2E 中文社区（Flash/EEPROM、WWDT、NONMAIN）、立创开发板 MSPM0 PID Wiki、VOFA+ 官方文档（JustFloat/FireWater）、CSDN 多篇时间片调度/匿名协议/编码器配置实测帖、ARM/CMSIS NVIC 文档（M0+ 2bit 4 级优先级）。

## 🔑 关键要点（速记）

- 分层固定为 BSP/HAL→Driver→Middleware→App：App 禁止直接调 DL_* 寄存器函数，Middleware/App 与芯片无关，这样 STM32 练手与 MSPM0 主力可共用上层、只换 BSP/Driver，移植成本最低。
- 电赛小车/循迹首选'前后台+时间片轮询'，不上 FreeRTOS：任务<15、周期固定、无复杂阻塞，裸机更易调试与临场加任务；只有多个长耗时阻塞任务并存才考虑 RTOS。
- 调度器用 1ms tick + 结构体数组(函数指针+周期+就绪标志)：中断里只置位不干活，慢任务在主循环跑；控制环(1~2ms PID+编码器)直接放定时器中断保证确定性。
- 比赛流程用表驱动 FSM(出发→循迹→识别→执行→返回→停车)，状态切换与动作分离；视觉只产生事件/结果，由 FSM 用'连续 N 帧一致'采信，防 K230 误识别。
- MSPM0G3507 是 Cortex-M0+，NVIC 只有 2 位优先级=4 级(0 最高)、无子优先级：控制环/编码器设 0、串口 RX 设 1、SysTick 设 2、VOFA 发送设 3。
- 参数掉电保存用 Flash 模拟 EEPROM：放低 32KB 区(10 万次擦写)，DL_FlashCTL 擦除(1KB/扇区，8 字节对齐)再写，加 magic+CRC 校验回退默认值；严禁误写 NONMAIN 区(会导致无法烧录)。
- VOFA+ JustFloat = 小端 float 数组 + 固定 4 字节帧尾 0x00,0x00,0x80,0x7F，是多通道 PID 在线整定神器；FireWater(ASCII)用于可读调试，并可串口回写指令免烧录改参。
- 看门狗喂狗放最低优先级慢任务(100ms)、绝不在中断盲喂；上电自检顺序：时钟/复位→Flash 参数 CRC→IMU/编码器→K230 握手→OLED 显示，失败进 ST_ERR 报警。
- 本地无电脑调参靠 OLED 菜单 + 三键/编码器旋钮(小 FSM 管菜单层级)，改完写 Flash；日志分级 LOG_E/W/I/D，发布版关掉 DEBUG 省带宽。
- Git 协作：.gitignore 屏蔽 Keil/CCS 编译产物，按模块分人(电控/视觉/上位机)从根上避冲突，每天先 pull/rebase，比赛节点打 tag(v1.0-track-ok)可秒回退。
