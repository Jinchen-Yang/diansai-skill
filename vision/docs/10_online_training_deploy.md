## 本模块做什么

`demos/10_online_training_deploy/main.py` 是上述亚博例程的原样副本。它读取：

```text
/sdcard/mp_deployment_source/deploy_config.json
/sdcard/mp_deployment_source/<训练平台导出的 .kmodel>
```

并自动取得类别表、模型输入尺寸、anchors、模型类型、置信度阈值和 NMS 阈值。当前操作者已确认该例程可在本机亚博 K230 上直接运行。

它与 `vision/reference/yolo_coco_pipeline/` 中的固定 COCO YOLOv8 参考代码不能混用：本模块对应训练平台的 `AnchorBaseDet` 模型和配置文件。

## 部署步骤

1. 在训练平台完成训练并下载 `mp_deployment_source.zip`。
2. 解压后，在 K230 SD 卡上创建 `/sdcard/mp_deployment_source/`。
3. 将 `.kmodel` 与同目录的 `deploy_config.json` 放入该文件夹。
4. 打开并运行本模块的 `main.py`。当前代码已使用 `display_mode = "lcd"`。

不要把在线训练平台导出的 `.kmodel` 填入固定 COCO YOLOv8 参考代码。两者的标签、输入尺寸、anchors 和后处理接口不一致。

## 现场标定参数（比赛时必看）

本模块的模型参数来自 SD 卡上的 `deploy_config.json`，不是 `main.py` 顶部常量。每次重新训练、下载新模型或现场调参后，首先检查这个文件。

| 参数 | 含义 | 什么时候调 |
| --- | --- | --- |
| `confidence_threshold` | 最低检测置信度；低于该值的候选框丢弃 | 误检较多时提高；漏检较多时降低。示例初始值为 `0.25`。 |
| `nms_threshold` | 同类重叠框的 NMS 阈值 | 同一物体有多个重叠框时降低；相邻两个真实物体被压掉时提高。示例初始值为 `0.40`。 |
| `categories` | 类别名称和顺序 | 由训练平台导出，不能手改类别顺序；它必须与训练时的标签一致。 |
| `img_size` / `anchors` / `model_type` | 模型输入与后处理的结构参数 | 由训练平台导出。更换 `.kmodel` 时必须与其配套的 JSON 一起替换，通常不在现场手动调。 |

调试顺序：确认 `.kmodel` 与 JSON 来自同一次训练 -> 检查 `categories` 是否正确 -> 先调 `confidence_threshold` -> 再调 `nms_threshold` -> 最后在不同距离、角度、光照下做实物验证。

## 采集训练图片（比赛时必看）

训练图片必须覆盖目标在真实比赛中可能出现的变化，不能只围绕同一张桌面、同一高度连续拍摄。

- **光照**：正常室内光、较暗光、强光/反光、侧光；避免所有图片曝光完全相同。
- **相机高度与距离**：低、中、高三个高度，并覆盖目标较近、正常工作距离、较远的情况。
- **观察角度**：正视、左/右斜视、俯视、轻微仰视，以及目标旋转后的姿态。
- **位置与背景**：目标在画面中心、边缘、部分截断；背景应包含赛场可能出现的干扰物。
- **清晰度**：保留少量真实的轻微模糊样本，但不能大量加入完全失焦、过曝到无法辨认或目标太小的图片。

每张图中出现的所有目标都必须标注；同一类别的框选规则必须一致。采集后先抽查标注框是否包含完整可见目标，再上传训练。

## 这是不是 YOLOv 几？

当前导出配置只写明：

```json
"model_type": "AnchorBaseDet"
```

它表示三组 anchors 的锚框式目标检测模型，而不是明确的 `YOLOv5`、`YOLOv8` 等版本标识。仅凭这个部署包不能严谨地称它为某一代 YOLO；更不能把它与仓库中固定的 YOLOv8 COCO Demo 混为同一个模型。

## 是否现在学习本地训练

当前不必暂停在线平台路线去配置 Ubuntu、本地 Conda、GPU 训练和 K230 模型转换。你已经完成了更重要的第一步：能独立完成在线训练并在板端运行。

建议在完成一次自己的 `watch` 模型板端实测后，再学习本地训练。这条路线在正式比赛时仍值得掌握，原因是可离线训练、可控制训练参数，也能避免云端排队；但它是第二阶段能力，不是当前部署在线模型的前置条件。
